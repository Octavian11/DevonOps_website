Implementation Spec

Typography & Hierarchy

Standardize the type scale across all pages. Fix heading weights for Garamond elegance, eliminate h3 size variations (currently 4 different sizes), tighten body text line-height, and create a consistent label system. All changes are global CSS.

5 implementation items April 2026
Diagnosis
Current typography audit findings
The Garamond + sans-serif pairing is strong. The issues are inconsistency in sizing, weight, and spacing — not the typeface choices themselves.
Critical 4 problems found across all pages

Problem 1: Garamond headings use weight 700. EB Garamond at weight 700 loses its characteristic elegance — the hairline/thick stroke contrast that makes serifs feel premium gets flattened. Weight 400 or 500 is the sweet spot for Garamond display use. The current 700 makes headings feel heavy and slightly generic.

Problem 2: h3 has 4 different sizes. Live DOM audit found h3 elements at 22px, 21px, 20px, and 18px — four distinct sizes with no apparent logic. FAQ questions are 21px/700, sub-headings are 22px/400, section sub-titles are 18px/600, and card headings are 20px/700. These should collapse to 2 sizes maximum.

Problem 3: Body text line-height is too open. Some paragraphs have generous line-height that works for long-form reading but feels loose in short card descriptions and bullet points. The same line-height shouldn't apply to a 3-line paragraph and a single-line list item.

Problem 4: Uppercase label styling is inconsistent. The hero category label ("OPS DILIGENCE · VALUE CREATION PLANS · POST-CLOSE GOVERNANCE") is 19px/400/1.5px tracking. Category pills are 15px/700/0.6px. Pricing tier labels are different again. These should all share a single label style.

Item 02
Heading weight — 400 for Garamond elegance
The single most impactful typography change. EB Garamond at weight 400 has beautiful thick/thin stroke contrast that signals "premium advisory." At 700, those strokes flatten and the font looks like any bold serif.
Critical Reduce h1 and h2 weight from 700 to 400
ElementCurrent weightNew weightWhy
h1 700 400 At 48px display size, weight 400 has commanding presence without heaviness
h2 700 400 At 28px, weight 400 reads clearly while maintaining serif refinement
h3 600–700 (varies) 600 Sub-headings need slightly more weight to anchor card content — 600 is the compromise
FAQ questions 700 600 FAQ rows at 700 feel heavy for an accordion — 600 is sufficient for scannability
Design rationale: Look at any premium advisory firm's website — McKinsey, Bain, KKR, Lazard. None of them use bold serif headings. They all use light or regular-weight serifs at display sizes. Weight 700 on a serif font is a tell that says "template" to a trained eye. Weight 400 says "designed." Since your buyers are PE professionals who see dozens of advisory pitch decks, this subtlety registers at a subconscious level.

Item 04
Body text line-height tightening
The current 1.7 line-height works for long-form paragraphs but feels loose in card contexts and list items. Different containers need different line-heights.
High Context-appropriate line-heights
ContextCurrent lhNew lhRationale
Section lead paragraphs 1.7 1.65 Slight tightening — still comfortable for 2-3 line descriptions
Card body text 1.7 1.6 Cards are dense by nature — tighter lh prevents wasted vertical space
List items 1.7 1.5 Bullet lists should be scannable, not airy
Blockquotes / proof 1.7 1.55 Quotes are short — tight lh increases visual weight of the statement
Hero subheadline ~1.6 1.55 The hero sub-text should feel compact and authoritative
/* ══ Context-specific line-heights ══ */ /* Section leads */ .section-wrapper > p:first-of-type, h2 + p { line-height: 1.65; } /* Card body */ .buyer-card p, .cost-card p, .pricing-card p, .path-card p, .outcome-card p, .track-vignette p { line-height: 1.6; } /* List items */ li { line-height: 1.5; } /* Blockquotes */ blockquote, .mckinsey-quote, .track-result, .buyer-card blockquote { line-height: 1.55; } /* Hero subheadline */ .hero-subheadline, .hero-block p { line-height: 1.55; }

Global scope
Cross-page application
All typography changes in this spec are global — they use element selectors (h1, h2, h3, p, li) that apply across every page automatically. The only page-specific selectors are hero-related classes.

Testing note: After deploying, the most visually noticeable change will be the h1 and h2 weight drop from 700 to 400. Garamond at 400 is noticeably lighter — this is intentional and correct. If any heading looks "too thin" at 400, the fix is NOT to go back to 700 — it's to go to 500 (which provides a middle ground). But 400 is the recommendation for maximum PE-appropriate refinement.
Implementation Summary
Priority sequence

Execute in this order

  • Pass 1 — Heading weights (Item 02): Change h1 and h2 from 700 to 400, h3 from varied to 600. This is the most impactful visual change — do it first and verify it looks right before proceeding.
  • Pass 2 — Type scale (Item 01): Standardize all h3 elements to 20px/600, FAQ to 18px/600. Fix the h3 inconsistency.
  • Pass 3 — Labels (Item 03): Apply the unified context-label style (11px/600/1.5px tracking) to all uppercase labels. Fix the hero category from 19px to 11px.
  • Pass 4 — Line-heights (Item 04): Apply context-specific line-heights to body text, card text, list items, and blockquotes.
  • Pass 5 — Mobile scale (Item 05): Apply the responsive type scale in the ≤768px media query.
